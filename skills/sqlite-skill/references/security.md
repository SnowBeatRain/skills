# SQLite 安全

SQLite 是本地数据库，不等于安全存储。端侧离线功能必须同时考虑 SQL 注入、数据库文件、敏感数据、日志和服务端权限校验。

## SQL 注入

必须使用参数绑定：

```ts
await db.query('SELECT * FROM users WHERE name = ?', [name])
```

禁止：

```ts
const sql = `SELECT * FROM users WHERE name = '${name}'`
```

动态列名、排序字段、表名不能直接来自用户输入，必须白名单映射。

## 敏感数据

默认不要明文存：

- 密码。
- access token / refresh token。
- 身份证、手机号、精确位置。
- 生物识别数据。
- 业务高敏数据。

如必须落库，优先考虑系统安全存储、字段级加密、SQLCipher 或原生加密插件。密钥不能硬编码在源码中，也不能存普通 storage。

## 数据库文件

SQLite 数据库是本地文件，可能被备份、root/越狱设备读取、调试工具导出或崩溃包间接泄露。

注意：

- SQLite wrapper 默认不等于加密数据库。
- 浏览器、移动端、桌面端或嵌入式本地存储都不能直接视为安全边界。
- 高安全业务需要服务端风控和最小化本地存储。

## 多账号隔离

切换账号时必须避免 A 用户看到 B 用户本地数据。策略：

- 每用户独立数据库文件；或
- 所有业务表加 `user_id`，所有查询都带当前用户条件。

退出登录时清理内存缓存、同步队列、敏感字段和附件缓存。

## 日志脱敏

生产日志不要输出完整 SQL 参数、payload、token、手机号、证件号、加密密钥。

## 服务端校验

离线同步上传的数据不可信。服务端仍需：

- 鉴权。
- 校验资源归属。
- 校验字段权限。
- 校验版本和冲突。
- 防重复提交。

客户端的 `user_id`、`role`、`price` 等字段不能作为服务端授权依据。

## SQLite 特有攻击面

- 默认禁止或严格白名单 `ATTACH DATABASE`，避免访问非预期文件。
- 数据库路径必须归一化并限制在应用允许目录内，防止路径穿越或覆盖文件。
- 外部导入 `.db` 文件时，先复制到隔离区，执行 `quick_check`、版本检查、表白名单校验，再决定是否导入。
- 不执行不可信 SQL 脚本，不启用不必要的 `load_extension`。
- FTS 查询表达式、JSON path、排序字段和动态列名不能直接拼接用户输入；需要转义、白名单或受控 QueryBuilder。
